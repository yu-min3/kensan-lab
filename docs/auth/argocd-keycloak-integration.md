# ArgoCD ↔ Keycloak OIDC integration (Path B)

ArgoCD uses Keycloak realm `kensan` as its IdP through direct OIDC integration, without going through Dex — the same "app native auth" pattern as Vault / Grafana.

## Configuration

| Element | Location | Notes |
|---|---|---|
| Keycloak client `argocd` | created by `bootstrap/keycloak/setup.sh` | redirect URI: `https://argocd.platform.yu-min3.com/auth/callback` |
| client_secret | Vault KV `secret/security/argocd/oidc` | injected by bootstrap TF |
| Kubernetes Secret `argocd-oidc-secret` | generated by ESO (ExternalSecret) | namespace: `argocd` |
| `oidc.config` in `argocd-cm` | `kubernetes/argocd/values.yaml` | referenced as `$argocd-oidc-secret:client-secret` |

## Key part of values.yaml

```yaml
configs:
  cm:
    url: "https://argocd.platform.yu-min3.com"
    oidc.config: |
      name: Keycloak
      issuer: https://auth.platform.yu-min3.com/realms/kensan
      clientID: argocd
      clientSecret: $argocd-oidc-secret:client-secret
      requestedScopes:
        - openid
        - profile
        - email
      requestedIDTokenClaims:
        groups:
          essential: true
  rbac:
    policy.csv: |
      g, platform-admin, role:admin
      g, platform-dev, role:readonly
    scopes: '[groups]'
```

## Handling scopes and claims

Do **not** include `groups` in `requestedScopes`. In Keycloak, `groups` is a claim name rather than a scope; putting it in scopes gets rejected with "Invalid scopes: groups".

The realm-level `groups` client scope (attached to all clients by default; managed by `bootstrap/keycloak/setup.sh`) puts the `groups` claim on the id_token. ArgoCD receives it by requesting `groups: { essential: true }` in `requestedIDTokenClaims`.

## RBAC mapping

| Keycloak group | ArgoCD role |
|---|---|
| `platform-admin` | `role:admin` |
| `platform-dev` | `role:readonly` |

The built-in admin user (`admin.enabled: true`, the default) is kept as break-glass.

## Why Path B (direct integration)

Reasons for choosing this over going through Dex (Path A):

- Vault / Grafana already use the same "direct Keycloak integration" pattern; putting Dex in front of ArgoCD alone would break the symmetry
- Dex adds a second hop of configuration (argocd → dex → keycloak), which complicates debugging
- ArgoCD bundles Dex internally, but for an external IdP, pointing `oidc.config` directly at Keycloak means Dex never needs to run
