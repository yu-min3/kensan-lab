# Documentation Layout

Documentation in this repository is **split into layers with distinct responsibilities**. The responsibilities are fixed below so that "where should I write this?" has a mechanical answer. The ground rule: **never write the same fact in two places.**

> **Human-facing vs AI-facing is not "two zones" but "one source → two outputs"**: every fact is written once, in `docs/`. The MkDocs build produces two outputs from it — the human-facing **HTML site** (overviews, diagrams, narrative) and the AI-facing **`/llms.txt` (index) + `/llms-full.txt` (full text)** (auto-generated by `mkdocs-llmstxt`, following the [llmstxt.org](https://llmstxt.org/) standard). The site nav is for humans only, so no audience labels are used. `.claude/rules/` stays thin and references `docs/`; agent-facing work instructions live in the root `AGENTS.md` / `CLAUDE.md` (facts are never duplicated).

## The six layers

| Layer | Role | Location | What to write | What NOT to write |
|---|---|---|---|---|
| **Top README** | Reference portal | `/README.md` | The big picture, hardware, entry links into each layer, the outward-facing "what is this" | Detailed how-tos, decision rationale, component specs |
| **Per-directory README** | Co-located what / where | `kubernetes/<cat>/README.md`, `kubernetes/<cat>/<comp>/README.md` | What this directory is, which components live here, where in docs/ to dig deeper | Cross-cutting background, decision rationale that belongs in an ADR |
| **`docs/` (MkDocs SoT)** | The human-facing SoT for cross-cutting how / why | `docs/<topic>/` | Detailed matrices, workflows, diagrams, explanations spanning multiple components | Decision history (link to ADRs), file layout conventions |
| **`docs/adr/`** | **Why** (immutable decision records) | `docs/adr/NNN-*.md` | Decision context, rejected alternatives, trade-offs, consequences | Current file paths, how-tos (ADRs are frozen in time) |
| **`.claude/rules/`** | **Thin AI-facing summaries + globs** | `.claude/rules/<topic>.md` | 3–5 line rules, applicable globs, links into `docs/` | Detailed tables (put them in docs/), human-facing prose |
| **`CLAUDE.md`** | **AI entry point** | `/CLAUDE.md` | Mandatory constraints, index into rules/, directory tree | Tech stack tables (Top README), component details (per-dir READMEs) |

## Deciding where a fact goes

When in doubt, walk this flow:

```
Is it "the reason for a decision"?
  └─ Yes → ADR (docs/adr/)
  └─ No  → Is it "an explanation fully contained within one directory"?
            └─ Yes → that directory's README.md
            └─ No  → Is it "cross-cutting how / why spanning multiple components"?
                      └─ Yes → docs/<topic>/
                      └─ No  → Is it a glob-scoped rule an AI needs to be told?
                                └─ Yes → .claude/rules/<topic>.md
                                └─ No  → it probably doesn't need writing at all
```

## README writing guide

`kubernetes/<cat>/README.md` and `kubernetes/<cat>/<comp>/README.md` must satisfy:

- **Length**: roughly 5–30 lines. Growing past that is the signal to push content out into `docs/`
    - **Exception — architecture-category READMEs** (`kubernetes/<cat>/README.md` files that serve as the transclude source of `docs/architecture/<cat>.md`): since they double as the SoT for the site's architecture pages, **up to ~150 lines plus mermaid diagrams is acceptable**. Follow the network README as the template (Design thesis → Components → diagram → Design rationale → Related)
- **Required sections**:
  - One-line summary (what is this directory / component)
  - The subdirectories / key files it contains
  - Deep-dive links (related docs/, ADRs, external URLs)
- **Forbidden**:
  - Empty templates full of TODOs
  - Detail tables duplicated from elsewhere (link instead)
  - "Why" explanations (those belong in ADRs)

### Minimal component README template

```markdown
# <component>

<one-line summary — why does this exist>

## Layout

- `values.yaml` — key Helm chart overrides (highlights only)
- `resources/` — extra manifests outside the chart (HTTPRoute, SealedSecret, …)

## Related

- Deep dive: [docs/<topic>/index.md](../../../docs/<topic>/index.md)
- Decision history: [ADR-NNN](../../../docs/adr/NNN-*.md)
```

## Drift prevention

Making each SoT unique prevents drift structurally. No CI checks or linting needed (the duplication simply doesn't exist).

Still, watch for:

- **Always come back to this page to pick a home before writing a new fact**
- **When editing an existing document, suspect that "the same fact probably exists somewhere else"**
- On finding a violation, consolidate into the SoT promptly and replace the other copies with links
