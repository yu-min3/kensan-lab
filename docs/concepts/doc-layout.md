# Documentation Layout

Documentation in this repository is **split by layer, each with a fixed responsibility**. The goal is that "where do I write this?" is a mechanical decision, not a judgment call. The basic rule: **never write the same fact in two places**.

> **Human-facing and AI-facing aren't "two zones" — they're "one source → two outputs"**: every fact is written exactly once, in `docs/`. The MkDocs build produces two outputs from that one source — a human-facing **HTML site** (overviews, diagrams, narrative) and an AI-facing **`/llms.txt`** (index) **+ `/llms-full.txt`** (full text), auto-generated by `mkdocs-llmstxt` (the [llmstxt.org](https://llmstxt.org/) standard). The docs site's nav is human-only, so no audience label is attached to anything. `.claude/rules/` stays thin and points at `docs/`; agent-facing task instructions live in the root `AGENTS.md` / `CLAUDE.md` (facts are never duplicated).

## The five layers

| Layer | Role | Location | Contains | Does not contain |
|---|---|---|---|---|
| **Top README** | Reference portal | `/README.md` | The big picture, hardware, entry links into every layer, the outward-facing "what is this" | Detailed how-tos, decision rationale, component specs |
| **Per-directory README** | Co-located what / where | `kubernetes/<cat>/README.md`, `kubernetes/<cat>/<comp>/README.md` | What this directory is, which components live here, where to dig deeper in `docs/` | Cross-cutting background, decision rationale (belongs in an ADR) |
| **`docs/` (MkDocs source of truth)** | The **human-facing source of truth** for cross-cutting how / why | `docs/<topic>/` | Detailed matrices, workflows, diagrams, explanations spanning multiple components | The history of decisions (link to the ADR instead), file-layout conventions |
| **`docs/adr/`** | **Why** (immutable decision records) | `docs/adr/NNN-*.md` | The context behind a decision, rejected alternatives, trade-offs, Consequences | Current file paths, how-tos (these get stale) |
| **`.claude/rules/`** | **Thin AI-facing summary + globs** | `.claude/rules/<topic>.md` | 3–5 line rules, the applicable glob, a link into `docs/` | Detailed tables (belongs in `docs/`), human-oriented explanation |
| **`CLAUDE.md`** | **The AI entry point** | `/CLAUDE.md` | Mandatory constraints, an index into `rules/`, the directory tree | The tech-stack table (belongs in the Top README), layout details (belongs in the per-dir README) |

## Topic → source-of-truth map

The definitive answer to "where is this information documented." When adding new information, use this table first to decide where it goes.

| Topic | Source of truth | Where to find a summary |
|---|---|---|
| Per-domain architecture (argocd / auth / network / secrets / storage / observability / backstage) | `kubernetes/<cat>/README.md` (**Model A**: `docs/architecture/<cat>.md` transcludes it. `storage` includes R2 backups, RecurringJobs, and StorageClasses) | The corresponding topic under `.claude/rules/` |
| Secret management (the matrix of the 4 methods, inventory, operational procedures) | `docs/secret-management/index.md` (architecture and design rationale live in `kubernetes/secrets/README.md` — a deliberate split of responsibilities) | `.claude/rules/security-secrets.md` |
| Backstage Golden Path (procedures for App Developers) | `docs/guides/backstage-golden-path.md` | `kubernetes/backstage/README.md`, `.claude/rules/environment-separation.md` |
| Namespace naming | `docs/adr/006-namespace-naming.md` (why) + `.claude/rules/environment-separation.md` (how) | `kubernetes/namespaces/README.md` |
| Helm multi-source layout convention | `kubernetes/README.md` (Pattern A/B) | `.claude/rules/helm-multisource.md` |
| GitOps workflow | `.claude/rules/gitops-workflow.md` | `CLAUDE.md` Mandatory Constraints |

| Tech stack | Top `README.md` | `docs/index.md` |

## How to decide where something goes

When unsure where a fact belongs, walk through this flow:

```
Is it "the reason behind a decision"?
  └─ Yes → an ADR (docs/adr/)
  └─ No  → Is it "an explanation self-contained within one specific directory"?
            └─ Yes → that directory's README.md
            └─ No  → Is it "cross-cutting how / why spanning multiple components"?
                      └─ Yes → docs/<topic>/
                      └─ No  → Is it a rule the AI needs, scoped by a glob?
                                └─ Yes → .claude/rules/<topic>.md
                                └─ No  → It probably doesn't need to be written down at all
```

## README writing guide

`kubernetes/<cat>/README.md` and `kubernetes/<cat>/<comp>/README.md` should satisfy the following.

- **Length**: roughly 5–30 lines. Growing past that is a signal to push the content out to `docs/`
    - **Exception — architecture-category READMEs** (the `kubernetes/<cat>/README.md` files that are the transclude source for `docs/architecture/<cat>.md`): since these double as the site's architecture-page source of truth, **up to ~150 lines plus a mermaid diagram is acceptable**. Use `network` as the template (Design thesis → Components → diagram → Design rationale → Related)
- **Required sections**:
  - A one-line summary (what this directory / component is)
  - A description of the subdirectories / key files it contains
  - Deep-dive links (related `docs/`, ADRs, external URLs)
- **Forbidden**:
  - An empty template with only TODOs
  - Detail tables that duplicate content living elsewhere (link to it instead)
  - "Why" explanations (those belong in an ADR)

### Minimal component README template

```markdown
# <component>

<one-line summary — why this exists>

## Layout

- `values.yaml` — the key Helm chart overrides (just the highlights)
- `resources/` — additional manifests outside the chart (HTTPRoute, SealedSecret, …)

## Related

- Deep dive: [docs/<topic>/index.md](../../../docs/<topic>/index.md)
- Decision history: [ADR-NNN](../../../docs/adr/NNN-*.md)
```

## Preventing drift

Making each fact's source of truth unique prevents drift structurally. No CI check or linter is needed — because duplication itself doesn't exist.

That said, keep the following habits:

- **Come back to this page to decide placement whenever you write a new fact**
- **When editing existing docs, get in the habit of suspecting "this fact probably exists somewhere else too"**
- If you find a violation, consolidate it into the source of truth promptly and replace the other spot with a link
