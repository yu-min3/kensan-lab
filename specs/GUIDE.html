SDD 開発ガイド（infra）

このリポジトリの infra（GitOps / kubernetes/）を Spec-Driven Development で開発する実践ガイド。
「仕様を書く → AI が worktree で自律実装 → 必ず検証 → draft PR」を 4 コマンドで回す。

/sdd-spec · /sdd-plan · /sdd-tasks · /sdd-impl

設計の背景は sdd-overview.html ／ ワークフロー定義は README.md。

2. 誰が何をやるか

3. 前提

  • 作業場所: infra はリポジトリ root で作業（Claude を root にルート）。apps/kensan/ 配下に居ると app 用スキルが有効になり infra の /sdd-* は出ない。
  • 必要ツール: helm / kubeconform / kubectl / argocd CLI。yamllint は任意。
  • cluster 接続: 無くても静的ゲートまで完走。あればライブ検証（server dry-run）まで走る。

4. ステップ・バイ・ステップ

例として「Loki に alert ルール評価を足す」を作る。

完了の定義は spec が持つ: /sdd-impl が「止まる前に必ず満たすゴール」として参照するのは、作った spec.md の ## Acceptance criteria ＋ tasks.md の ## Verification。これを Definition of Done として取り込み、全項目を満たす（or cluster 未到達で merge 後確認、と理由付きで仕分ける）まで完了にしない。spec がそのまま停止ゲートの中身になる。

Step 0（任意）— 組み込み /goal を番人にする

組み込み /goal は同じ「停止前確認」を harness 層で行う別レイヤー。スキルからは自動セットできない（手で打つ UI 機能）ので、/sdd-impl が Step 4 でそのまま貼れる /goal 文字列を提示する。先回りして自分で打ってもよい:

/goal 001 の静的ゲート全緑 + spec の受入基準充足 + draft PR 作成

貼れば harness 側でも「検証未達のまま停止」を防げる。貼らなくても /sdd-impl 自身の DoD ゲートで同じ規律を守るので必須ではない。

Step 1 — 要件を固める /sdd-spec

/sdd-spec loki-ruler "Loki に alert ルール評価を足したい"
  • AI が目的・対象 namespace・公開有無・セキュリティ影響・受入基準を 2〜4 問質問 → specs/001-loki-ruler/spec.md 生成。
  • what / why だけ。chart 名や version は書かない。不明点は [NEEDS CLARIFICATION] で残り、残る間は次に進めない。
  • 👤 レビュー: 受入基準が「確認できる形」か。これが後の検証の物差し。

Step 2 — 技術設計 /sdd-plan

/sdd-plan loki-ruler
  • AI が kubernetes/ の似た既存を読み plan.md を作る: 配置（Pattern A/B/ApplicationSet）・chart repo・固定 version・Argo CD project・namespace・Gateway・Secrets 方式・## Affected paths。
  • 👤 レビュー: chart/version・namespace・secrets 方式が妥当か。## Constitution check で既存ルール準拠か。

Step 3 — タスク分解 /sdd-tasks

/sdd-tasks loki-ruler
  • spec + plan から依存順チェックボックス tasks.md を生成。末尾に必須の検証タスクが必ず入る。
  • 👤 レビュー: 抜け・順序を確認。

Step 4 — 自律実装 + 検証 + draft PR /sdd-impl

/sdd-impl loki-ruler

/sdd-impl は次を自律でやる:

偽 pass を許さない — 静的ゲートが全緑になるまで「完了」と言わない／cluster 未到達の検証は黙ってスキップせず PR body に明記／ゲート回避のための握り潰し禁止（直せなければ PR を作らず人間にフラグ）。

Step 5 — レビュー → ready → merge（👤 あなた）

draft PR を GitHub で確認。PR body に検証サマリ・受入基準の仕分け・merge 後 runbook が入っている。問題なければ ready 化 → merge。

Step 6 — 反映確認（👤 あなた）

/argocd-sync loki-ruler     # Synced + Healthy
/cluster-status             # pod / gateway / cert

merge すると Argo CD が自動 sync。「merge 後でないと確認できない」受入基準をここで確認する。

5. 検証はなぜ 2 段階か

infra には make test のようなテストスイートが無く、本番の正しさは merge 後の Argo CD が決める。だから分けている。

段階いつ何を掴めること
静的ゲートmerge 前・常時yamllint / helm template / kubeconform / dry-run=client / 安全 check構文・スキーマ・レンダリング・version 固定・secret 漏れ
ライブ検証merge 前・cluster 到達時kubectl dry-run=server実 API・CRD・admission（非破壊）
Argo CDmerge 後/argocd-sync /cluster-status /cert-checkSynced+Healthy・pod 起動・cert 発行
検証目的の一時的な kubectl --dry-run=server や scratch namespace への一時 apply→即 delete は許容。ただしデプロイ自体は必ず Git 経由（GitOps）。

6. Tips / FAQ

  • やり直したい: 前段コマンドを再実行して md を上書き、再レビュー。下流（plan/tasks）は上流変更後に作り直す。
  • worktree の片付け: PR merge 後に git worktree remove ~/kensan-lab.worktrees/<NNN-slug>。
  • cluster が無い環境: 静的ゲートまで完走。ライブ検証は「未実行」と明示され merge 後の Argo CD に委ねる。
  • secrets: SealedSecret は temp/<name>-raw.yaml（git-ignored）→ kubeseal → resources/<name>-sealed.yaml。生ファイルは絶対 commit しない。
  • app の開発は？: app は apps/kensan/ に cd して作業。app 用 SDD は今後 apps/kensan/.claude/ + apps/kensan/specs/ に同じ /sdd-* 名で用意予定（検証は make test / npm run build / Playwright に差し替え）。
