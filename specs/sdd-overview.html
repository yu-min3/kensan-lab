0. なにを作ったか（要約）

root .claude/ に 4 つのスキルと specs/ 一式を追加。spec-kit の specify → plan → tasks → implement を、このリポジトリの GitOps 文脈に合わせて作り直したもの。

コマンドフェーズ入力 → 出力
/sdd-spec <name>1 要件対話 → specs/NNN-slug/spec.md（what/why）
/sdd-plan <name>2 設計spec.md → plan.md（how / Helm パターン）
/sdd-tasks <name>3 分解plan.md → tasks.md（依存順 + 必須検証）
/sdd-impl [name]4 実装tasks.md → worktree 実装 + 検証 → draft PR

spec-kit の constitution.md は作っていない。憲法の役割は既存の CLAUDE.md + .claude/rules/ がそのまま担う（後述）。

sdd- 接頭辞は、Claude Code 組み込みの /goal（"set a goal Claude checks before stopping"）や他プラグインとの衝突回避。/goal は別物（停止前にゴールを確認する番人）で、/sdd-impl と併用する（§6）。

1. SDD とは / なぜこの形にしたか

SDD（Spec-Driven Development）は「コードの前に構造化された仕様を書き、それを single source of truth にして AI に実装させる」やり方。 狙いは、要件を前倒しで形式化し、広いコンテキストウィンドウを活かして AI と協働する土台を作ること。

一方 Fowler の批判（markdown レビュー負担が増える / AI が詳細指示を無視する / 過剰なファイル構造が「制御している錯覚」を生む）も踏まえ、軽量さを最優先した:

取り入れた点

  • フェーズ分離（spec→plan→tasks→impl）で各段にレビュー可能な成果物を残す
  • 仕様 = chat ではなくファイルが正
  • 「必ずテストして確認する」最終フェーズ

あえて避けた点

  • 専用 CLI（specify init / .specify/）依存 → ネイティブ skill のみ
  • constitution.md の新規作成 → 既存 rules を参照
  • 巨大テンプレ → 各テンプレは 1 画面程度

4. 設計のキモ① — ディレクトリスコープで infra / app を分離

「infra と app はフローが別物」「app は apps/ 配下で完結」という方針を、Claude Code のスキルが cwd でスコープされる性質で実現。 同名コマンドを両層に置いても、有効になるのは現在地の側だけなので衝突しない。

5. 設計のキモ② — 憲法は複製しない

spec-kit は統治原則を constitution.md に新規生成する。だがこのリポジトリには既に階層化された統治ルールがある。 そこで plan.md の ## Constitution check でリンク参照するだけにし、二重管理を避けた。

GitOps / worktree 原則gitops-workflow.md
コンポーネント構造helm-multisource.md（3-file パターン）
secrets の作法security-secrets.md
ネットワーク / 公開network-ingress.md

6. 設計のキモ③ — /sdd-impl の自律実装 + 必須検証ループ

ここが「最後に必ずテストして確認するフェーズ」という要求の中核。 /sdd-impl は worktree で実装した後、検証ゲートが全緑になるまで完了を宣言しない。

完了の定義 (DoD) は spec が持つ: /sdd-impl は開始時に spec.md の ## Acceptance criteria ＋ tasks.md の ## Verification を DoD として取り込み、それを満たすまで完了にしない。作った spec がそのまま停止ゲートの中身になる。
組み込み /goal は同じ停止前確認の harness 層だがスキルからは自動セット不可なので、/sdd-impl は貼れる /goal 文字列を提示する（ユーザー任意）。

偽 pass を許さない 3 つの不変条件 —— ❶ 静的ゲートが全緑になるまで「完了」と言わない。 ❷ cluster 未到達でライブ検証を飛ばす時は黙ってスキップせず明示し runbook に委ねる。 ❸ ゲートを通すために検証を握り潰さない（直せなければ人間にフラグ）。

7. なぜ infra の検証は「2 段階」なのか

infra にはアプリのような make test が無く、本番の正しさは merge 後の Argo CD が決める。 そこで「merge 前にローカルで掴めるもの」と「merge 後にしか分からないもの」を切り分けた。

段階いつ何を掴めること
静的ゲートmerge 前・ローカル・常時 yamllint / helm template / kubeconform / dry-run=client / 安全 check 構文・スキーマ・レンダリング・version 固定・secret 漏れ
server dry-runmerge 前・cluster 到達時 kubectl apply --dry-run=server 実 API・CRD 存在・admission webhook（非破壊）
Argo CDmerge 後 /argocd-sync / /cluster-status / /cert-check Synced+Healthy・pod 起動・cert 発行・mTLS 等（実デプロイ）
ユーザー許可済みの一時適用: 検証目的に限り kubectl --dry-run=server や scratch namespace への一時 apply→即 delete を許容。 ただし「GitOps only（デプロイは Git 経由）」の原則は不変 — あくまで検証のためであり、これがデプロイ手段になることはない。

8. 一回の流れ（例）

# 1. 要件を対話で固める
/sdd-spec loki-ruler "Loki に alert ルール評価を足したい"
   → specs/001-loki-ruler/spec.md（質問に答えながら確定）

# 2. 技術設計
/sdd-plan loki-ruler
   → plan.md（chart·version·namespace·Gateway·secrets·Affected paths）

# 3. タスク分解
/sdd-tasks loki-ruler
   → tasks.md（依存順 + 末尾に必須検証タスク）

# 任意: 組み込み /goal を番人としてセット
/goal 001 の静的ゲート全緑 + 受入基準充足 + draft PR

# 4. 自律実装 + 検証 + draft PR（worktree 内）
/sdd-impl loki-ruler
   → ~/kensan-lab.worktrees/001-loki-ruler で実装
   → 静的ゲート緑 → push → gh pr create --draft → PR URL 提示

# 5. 人間が PR レビュー → ready 化 → merge
# merge 後
/argocd-sync loki-ruler   &&   /cluster-status
